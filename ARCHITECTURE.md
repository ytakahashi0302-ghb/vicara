## コアコンセプト

### スクラムを人間とAIの共通プロトコルとして使う

スクラムは「人間チームのためのフレームワーク」として生まれましたが、
vicara では**人間とAIが協働するための共通言語**として再解釈します。

- 人間は「何を作るか」と「どの道筋で進めるか」を決める
- AIは「どう作るか・どう検証するか」を担う
- スクラムのプロセスが、その協働の透明性と意思決定の一貫性を担保する

### AIロールの責務分離

vicara の AI は、ひとつの万能エージェントとして振る舞うのではなく、役割を分けたチームとして設計します。

- **POアシスタント**: プロダクトオーナーの判断を補佐する。要求整理、優先順位づけ、バックログ追加、スプリント進行判断の相談窓口を担う。
- **開発エージェント**: ロールテンプレートに従って Claude Code CLI を実行し、個々のタスク実装と検証を進める。
- **人間のPO/開発者**: 何を作るか、どこで止めるか、最終的に採用する変更は何かを決める。

## Tauri × AIエージェントのアーキテクチャ

AIエージェントがタスクを自律実行する際の処理境界を明確にします。

```text
フロントエンド（React）
│  カンバン表示・進捗確認・人間の操作
│
│  invoke("execute_task", { task_id, instructions })
│
Rustバックエンド（Tauri）         ← 実際の処理はここで行う
├── AI API呼び出し（reqwest）
├── ファイル操作（std::fs）        ← バリデーション後に実行
├── コマンド実行（std::process）   ← 許可リスト制限あり
└── SQLite読み書き（sqlx）
```

**設計方針：フロントエンドに直接のシェル実行権限は与えない。**
AIの指示をRust側で受け取り、バリデーションを挟んでから実行することで、
AIの誤操作によるファイル破壊などのリスクを防ぎます。

## AIエージェントプロセス実行基盤 (Epic 24)

外部ツール（Claude CLI 等）を自律実行するための実行基盤は、プラットフォーム間の互換性と安定性を考慮した設計となっています。

### 1. プラットフォーム別実行戦略
*   **Unix (macOS/Linux)**: `portable-pty` を使用した仮想ターミナル (PTY) 実行。ANSIエスケープシーケンスやカラー出力をフルサポートし、インタラクティブな出力を忠実に再現します。
*   **Windows**: ConPTY の制限を回避するため、`std::process::Command` と `Stdio::piped()` を組み合わせた標準パイプ方式を採用。安定したログ出力を担保します。

### 2. 生存管理とタイムアウト
*   **Process Session**: 実行中のプロセスハンドルは、専用の `ClaudeSession` 構造体として Tauri State に保持し、Rust 側の関数スコープを抜けてもプロセスが維持されるように管理されています。
*   **Guardrails**: 180秒の強制タイムアウトスレッドが、暴走や無限ループによるコスト肥大化を防止します。また、フロントエンドからの強制終了 (Kill) シグナルを `ProcessKiller` trait を通じて抽象化して処理します。

## 進行中アクションの安全機構 (Interaction Guard)

スプリントタイマーやAIエージェントなどの「進行中のアクション」が存在する状態で、ユーザーがワークスペース（プロジェクト）を切り替えようとした場合、データの不整合や意図しない裏側での処理を防ぐための「Interaction Guard（安全機構）」を設けます。

### 設計方針

フロントエンドのUIコンポーネント（例: `ProjectSelector`）は、プロジェクトを切り替える前に、現在アクティブな処理がないかを各Contextからチェックします。

1. **状態の検知**: `SprintTimerContext` 等を用いて、タイマーが `RUNNING` 状態であるかを判定。将来的にはAIエージェントの処理中 (`isExecuting`) なども検知対象に含めます。
2. **ユーザーへの確認**: アクティブな処理が検出された場合、直ちに切り替えるのではなく、確認ダイアログ（`window.confirm`等）を表示します。
3. **安全な停止と切り替え**: ユーザーが切り替えに同意した場合、進行中の処理を安全に停止（あるいは一時停止）してから、`currentProjectId` の変更を行います。同意しない場合は切り替えをキャンセルします。

この仕組みにより、バックグラウンドで不要な処理が別のプロジェクトコンテキストで走り続けることを防ぎます。
