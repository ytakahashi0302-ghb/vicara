# MicroScrum AI 開発ガイドライン (Rule.md)

## 1. プロジェクト概要
- **目的**: ローカル環境でセキュアに動作する、AIネイティブなスクラム開発カンバンツール。
- **コアドメイン**: 1スプリント=1時間～数時間の「マイクロスクラム」。Story（親）とTask（子）の強固な紐付けをGUIで可視化する。

## 2. 技術スタック
- **Frontend**: React 18+, TypeScript, Tailwind CSS
- **UI Library**: dnd-kit (カンバン機能用), lucide-react (アイコン用)
- **Backend/Desktop**: Tauri v2 (Rust)
- **Database**: SQLite (ローカルファイル)
- **DB Access**: `@tauri-apps/plugin-sql` およびRust側の `tauri-plugin-sql`

## 3. AI（Dev）へのコーディング規約
- **TypeScript**: 厳格な型定義（`any` の使用禁止）。インターフェースは `types/` ディレクトリ等に集約すること。
- **React**: 関数コンポーネントとHooksを使用すること。UIとビジネスロジックを適切に分離すること。
- **SQLite操作**: フロントエンドで直接複雑なSQLを組み立てるのではなく、極力Tauriのコマンド（Rust側）で安全に処理するか、プラグインの作法に則りSQLインジェクション対策（プレースホルダーの利用）を徹底すること。
- **状態管理**: まずはReact標準のContext/Stateで小さく始め、複雑化した段階で適切なライブラリ導入をPOに提案すること。
- **UIのリアクティビティ確保 (重要)**: DB（SQLite）やStoreへのデータ更新（Mutation）を行った際は、必ずセットで関連するReact Stateの同期（Refetch関数の呼び出し、またはローカルStateの更新）を実装し、ユーザーの画面（UI）が即座に最新状態になること（リアクティビティ）を徹底すること。
- **UIの基本言語とローカライズ**: エンドユーザーが触れるUI（ボタン、ラベル、プレースホルダー、トースト通知など）は、常に自然な日本語で実装すること。DBの保存値や内部のロジック（例: Statusの `To Do`, `In Progress`, `Done`）は英語のままで良いが、画面表示時は適切に日本語にマッピングすること。

## 4. AIエージェントの振る舞い（重要）
- **思考の透明性**: 実装前に必ずアプローチをPOに提示し、承認を得てからコードを書くこと。
- **破壊的変更の禁止**: 既存のコードを大きく書き換える場合や、新しいnpmパッケージ/Rustクレートを追加する場合は、必ず事前にPOへ理由を説明し許可を得ること。
- **コンテキストの維持**: チャットセッションを終了する際は、必ず現在の進捗と次やるべきタスクをまとめた `handoff.md` を作成すること。
- **テストと完了報告 (重要)**: 実装が完了し、POに手動検証（マニュアルテスト）を依頼する際は、必ず事前に（または同時に）`walkthrough.md` を作成・出力し、「変更内容」「テスト手順」「検証結果」を明示すること。
- **ドキュメントの更新（重要）**: 以下のタイミングで対応するドキュメントを必ず更新すること。更新を忘れるとドキュメントとコードの乖離が生じる。
  - `Rule.md`: コーディング規約・技術スタック・技術的負債に変化があった場合
  - `PRODUCT_CONTEXT.md`: 設計方針・やらないことの定義・プロダクトの思想に関わる判断をした場合
  - `ARCHITECTURE.md`: 機能設計・DB設計・エージェント設計・APIの仕様に変化があった場合
- **必須ファイルの参照（チャット開始時）**: 新しいチャットセッションを開始する際は、必ず最初に以下の2ファイルを読むこと。これはすべての実装・設計判断の前提となる。
  - `Rule.md`（このファイル）: How（どう作るか）
  - `PRODUCT_CONTEXT.md`: Why（なぜ作るか・設計判断の理由・やらないこと）
- **追加参照（設計判断が必要な場合）**: 機能設計・DB設計・エージェント設計に関わる判断をする場合は、追加で以下を参照すること。
  - `ARCHITECTURE.md`: What（何を作るか・詳細設計）

## 5. 命名規則・採番ルール
- **実装順序の可視化（採番ルール）**: 実装の順序やフェーズが後から見てわかるように、ドキュメント、各種計画ファイル、機能単位の作業ディレクトリ等を作成する際は、必ずプレフィックスとして採番（例: `01_setup/`, `02_frontend_db/`, `03_kanban_ui/`）を行うこと。
  - ※ただし、`src/components` のようなReact/Tauriの標準的なディレクトリ構造の命名作法は崩さない範囲で適用すること。

## 6. 技術的な知見と制約（Tips & Constraints）
- **Tailwind CSS v4**: Vite + Tailwind v4環境では `postcss.config.js` に `tailwindcss` ではなく `@tailwindcss/postcss` を指定する必要がある。また CSSファイルのエントリーポイントには `@tailwind base;` ではなく `@import "tailwindcss";` を使用する。
- **Tauri v2 SQL Permissions**: `@tauri-apps/plugin-sql` をフロントエンドから使用する際、`capabilities/default.json` にて `"sql:default"` を許可するだけでなく、実際にコマンドを叩くために `"sql:allow-execute"` および `"sql:allow-select"` も明示的に追加しなければならない。

## 7. 課題・技術的負債 (Tech Debt)
- **Inception Deck対話履歴の揮発性**: Inception Deckで一度画面を閉じるとチャット履歴が消えてしまい、再開時のコンテキストが分かりにくい（状態の永続化が未実装）。
- **Inception Deckへの再入線UI欠如**: カンバン画面等から「インセプションデッキを再度見直す・修正する」ための明確なUI動線（ルーティング）が存在しない。
- **タイマー稼働中の新規プロジェクト作成によるガード漏れ**: スプリントタイマーが稼働している状態（RUNNING）で、ヘッダーのドロップダウンから「新規ワークスペース作成」を選択して新しいプロジェクトを作成した場合、Interaction Guard（確認ダイアログ）が発動せずに状態が新プロジェクトに引き継がれてしまうバグが存在する。レアケースであるため現状は許容（Tech Debt）とし、次回以降の開発で修正を検討する。
- **AIタスク生成時のJSONパース正規表現の脆弱性**: 現在 `generate_tasks_from_story` 内のJSON抽出処理が最短マッチの正規表現 (`(?s)\[.*?\]`) で行われており、ネストされた配列などでパースが失敗する可能性がある。将来的にGeminiのように Structured Output (`application/json`) を Anthropic 側でも利用できるようにする。
- **WorkspaceContext の初期プロジェクトIDハードコード**: `currentProjectId` の初期値が `'default'` と固定されているため、これが削除された場合のフォールバック（最初の有効なプロジェクトを選択する等）が未実装である。
- **AIモデルバージョンのハードコード**: Rust側 (`ai.rs`) において、Gemini,Anthropicのモデル名が決め打ちになっている。次回以降、設定画面（Settings）から利用モデルを選択できるような仕組みを導入する。
- **CRUD操作時の全件再フェッチ**: ストーリー等の追加・更新時に現状は `fetchStories()` 等でリストを全件再取得している。データ量増加時に遅延やチラつきの原因となるため、カンバン等で導入済みの楽観的UI (Optimistic UI) への段階的移行を検討する。
- **未使用コード・デバッグログの残留**: Tauriテンプレート由来の `greet` コマンドや、`db.rs` 内の `println!` が残っている。リリース前にクリーンアップする。
- **TauriコマンドのContext未公開**: `update_project` や `delete_project` が WorkspaceContext 経由で呼び出せるようになっていないため、呼び出し経路が統一されていない。
- **プロジェクト削除機能のUI欠如**: バックエンドに `delete_project` コマンドがあるが、フロントエンドからそれを呼び出してプロジェクトを削除するUI機能が未実装である。
- **AI呼び出しコスト(予算・トークン)の未計測**: `FUTURE_CONCEPT.md` で言及されているコスト管理を実現するために、AI呼び出しごとの usage(input/output tokens) を記録する機能を追加し、さらにチャット単位でリアルタイムにコスト使用量を画面上で確認できるUIを実装する。
- **設定ボタンのアクセシビリティとUIレイアウト課題**: 現在、設定ボタンの配置が最適ではなく、特定の画面からしかアクセスできない状態となっている。カンバンボードやInception Deckなど、アプリ内のどこからでもシームレスに設定画面へアクセスできるよう、共通のグローバルナビゲーション（ヘッダーやサイドバー等）への配置見直しを行う。
- **インセプションデッキのフェーズ移行のUI化未済**: 現在、インセプションデッキのチャット進行において、特定の会話（デッキの1項目など）が完了し次のフェーズへ進む際、ユーザーが都度手動で「次へ進めて」とテキスト入力する必要がありUXを損ねている。自然言語での指示に依存せず、ボタン押下（例：「次のフェーズへ進む」ボタン）によって明確にフェーズの状態遷移（ステート更新およびAIへのシステムプロンプト/コンテキスト送信）をトリガーできるUI・制御ロジックを実装する。
- **Claude APIの応答特性（Chattyさ）によるJSONパースエラー**: インセプションデッキ作成等のAI対話処理において、GeminiとClaudeで応答フォーマットに対する挙動に差異がある。Claude（claude-3-5-sonnet等）を使用した場合、JSON形式での出力を期待するコンテキストであっても、自然言語による前置きや会話テキストのみを返す傾向があり、フロントエンド（InceptionDeck.tsx）でのJSONパースに失敗する（Failed to parse JSON: expected value at line 1 column 1）。システムプロンプトでの出力フォーマットの厳格化（「JSON以外は絶対に出力しないこと」などの強い制約）、またはAnthropicの Tool use (Function calling) を活用して、確実に構造化データを抽出・処理できる堅牢な仕組みへの改修が必要である。
- **LLMの特性差異によるドキュメント全上書きバグ**: インセプションデッキ生成プロセスにおいて、Claude APIを使用すると PRODUCT_CONTEXT.md などのドキュメント更新時に、既存のセクション（例：Phase 1のWhy）が新しいセクション（例：Phase 2のNot List）によってファイルごと上書き・消失してしまう問題が発生している。Geminiでは発生しなかった挙動であり、Claudeの「現在のタスクフォーカスのみでドキュメントを再構築する」特性に起因する。システムプロンプトにおいて「既存のファイル内容を読み込み、失うことなく特定のセクションのみを追記・更新すること」を厳格に指示するか、バックエンド側のファイル書き込みロジックを全置換ではなくマージ（差分更新）型に改修する必要がある。
- **AIによるタスク自動生成ボタンの消失**: 以前のアクティブスプリントベースのカンバン画面に実装されていた「ストーリーからAIが子タスクを自動生成する（generate_tasks_from_story）」ボタンが、Epic 6の大規模な基盤改修（Backlog分離）の際にバックログ画面(`BacklogView.tsx`)へ再配置されずに消失してしまっている状態（実装漏れ）。次回のAIエンハンス計画などで復旧・配置見直しを行う。
