0. 設計ゴール（明確化）

この基盤は以下を満たす必要があります：

AI（Devエージェント）が環境を継続的に操作できる
POエージェントが監査・停止・介入できる
出力が逐次ストリームされ意思決定に使える
OS差異を安全に吸収
将来的に SSH / コンテナ / リモート実行に拡張可能

2. コア設計（最重要）
2.1 Commandは「文字列禁止」
pub struct CommandSpec {
    pub program: String,
    pub args: Vec<String>,
    pub cwd: Option<PathBuf>,
    pub env: HashMap<String, String>,
}

👉 LLMに文字列を作らせない
👉 Shell injection防止
👉 OS差異吸収

2.2 出力は「意味単位ストリーム」
pub enum OutputEvent {
    StdoutLine(String),
    StderrLine(String),
    SystemMessage(String),
    Exit { code: i32 },
}

👉 byte列禁止
👉 行単位でAIが解釈可能

2.3 ExecutionResult（単発用）
pub struct ExecutionResult {
    pub exit_code: i32,
    pub stdout: Vec<String>,
    pub stderr: Vec<String>,
}
3. 実行モデル（重要）
3.1 明示的に分ける
pub enum ExecutionMode {
    OneShot,
    Streaming,
    Interactive,
}
3.2 トレイト設計
#[async_trait]
pub trait CommandExecutor: Send + Sync {
    async fn exec_one_shot(
        &self,
        cmd: CommandSpec,
    ) -> Result<ExecutionResult>;

    async fn exec_stream(
        &self,
        cmd: CommandSpec,
    ) -> Result<tokio::sync::mpsc::Receiver<OutputEvent>>;

    async fn exec_interactive(
        &self,
        session_id: SessionId,
        cmd: CommandSpec,
    ) -> Result<()>;
}

5. ストリーム設計（超重要）
5.1 bounded channel
let (tx, rx) = tokio::sync::mpsc::channel(100);

👉 backpressure制御

5.2 Readerタスク（標準構成）
tokio::spawn(async move {
    let reader = BufReader::new(stdout);
    let mut lines = reader.lines();

    while let Ok(Some(line)) = lines.next_line().await {
        if tx.send(OutputEvent::StdoutLine(line)).await.is_err() {
            break;
        }
    }
});
5.3 stderrも同様（ただし統合）

👉 順序問題対策：

シンプル：stdout/stderr分離（PoC）
本番：pty or merged pipe

10. 拡張性（重要）

この設計はそのまま拡張可能：

機能	拡張方法

並列実行	Session分離
権限制御	CommandSpecに追加
11. 今回の指示との差分（重要）
項目	指示	本設計
コマンド	文字列	構造体
出力	byte/曖昧	意味単位
抽象	単一trait	実行モデル分離
セッション	なし	明示的
監査	なし	CommandLog
OS差分	隠蔽	明示
AI前提	弱い	強い
