# 調査レポート：マルチエージェント・オーケストレーションの方式

## 1. 概要
AIエージェントを単体で動かすのではなく、複数の専門エージェントを連携させて複雑なタスクを解決する「マルチエージェント・システム（MAS）」の主要な方式と、本プロジェクトへの適用可能性について調査した。

## 2. 主要フレームワーク比較 (2025年時点)

| フレームワーク | 設計思想 | 特徴・強み | 主な用途 |
| :--- | :--- | :--- | :--- |
| **LangGraph** | 状態遷移図 (Graph) | 精密なフロー制御、ループ構造、全状態の永続化が可能。 | 本格的な業務フロー、複雑な条件分岐 |
| **CrewAI** | 役割ベース (Role-based) | Personaの設定が容易。各エージェントが自律的にタスクを分担。 | 記事執筆、マーケット調査、開発チーム擬似 |
| **AutoGen** | 会話型 (Conversational) | 非同期対話、多人数チャット形式。試行錯誤に強い。 | コーディングアシスタント、アイデア出し |
| **OpenAI Swarm** | 移譲型 (Handoffs) | 軽量・シンプル。特定のタスクが終わったら次の担当へ渡す。 | プロトタイピング、カスタマーサポート |

## 3. オーケストレーション・パターン

エージェント間の連携アーキテクチャには以下の代表的なパターンが存在する。

- **Sequential (直列型)**:
  - Agent A → Agent B → Agent C と順番に処理をバトンタッチする。
  - シンプルだが、後戻りが難しい。
- **Router / Dispatcher (司令塔型)**:
  - 1つの「司令塔」がユーザーの意図を汲み取り、適切な「専門エージェント」へ仕事を振る。
- **Hierarchical (階層型)**:
  - マネージャー層が全体の進捗を管理し、ワーカー層に具体的な作業を委任する。
- **Collaborative (協調型/スウォーム)**:
  - 複数のエージェントが対話を繰り返し、最終的な合意（結果）を導き出す。

## 4. プロジェクト（Tauri + Rust）への適用検討

MicroScrum AIの文脈において、以下の実装アプローチが考えられる。

### アプローチ1：Rust Native (Rig 等の利用)
- **特徴**: Rustバックエンドでエージェント・オーケストレーターを実装。
- **メリット**: アプリケーションの一部として高速に動作し、Tauriのコマンドと統合しやすい。型安全性が高い。
- **候補ライブラリ**: [Rig](https://rig.rs/)

### アプローチ2：TypeScript / LangGraph.js
- **特徴**: フロントエンド側でオーケストレーションを管理し、推論だけをRust経由でAPIに投げる。
- **メリット**: UIとの親和性が高く、JavaScriptエコシステムの豊富なMASライブラリを利用可能。

### アプローチ3：Python Sidecar
- **特徴**: 既存の強力なPythonライブラリ（LangGraph等）をSidecarとして同梱。
- **メリット**: 最先端のAI機能をそのまま使える。
- **デメリット**: 配布サイズが大きくなり、ランタイムの管理が複雑。

## 5. 結論と推奨

現在のプロジェクト設計（ソロ開発の効率化）においては、**「Routerパターン」または「Sequentialパターン」をRust側で軽量に実装する（案1）**のが、保守性とパフォーマンスのバランスが最も良い。

具体的には、`ai.rs` 内で以下の役割（エージェント）を定義し、それぞれのプロンプトと出力スキーマ（JSON）を定義することを推奨する。
- **PO Assistant**: 要求の深掘り、ストーリー作成
- **Architect**: 技術選定、タスク分解
- **Developer**: 実行、コード生成
- **Reviewer**: 成果物チェック
