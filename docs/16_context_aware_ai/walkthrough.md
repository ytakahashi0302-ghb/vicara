# Epic 3: コンテキスト・アウェアなAI要件定義 変更内容と確認手順（Walkthrough）

## 変更内容 (Changes)

- **フロントエンド側へのプロジェクトコンテキスト紐付け**
  - `IdeaRefinementDrawer.tsx` (AI要件定義アシスタント) および `StorySwimlane.tsx` (AIタスク生成) において、`WorkspaceContext` から動的に `currentProjectId` を取得し、バックエンドの `invoke` メソッドの引数として渡すよう改修しました。
- **バックエンド側 (Rust) での プロジェクトデータ取得ロジック（簡易RAG）の実装**
  - `src-tauri/src/db.rs` に `build_project_context` 関数を追加実装し、特定の `project_id` に属する有効なストーリー（`archived = 0`）とそれに関連するタスク一覧を一括取得する処理を追加しました。
  - プロンプトのトークン消費を抑えるため、Description は最大50文字程度で切り詰めるようにフォーマットを工夫し、シンプルで効果的なMarkdownの箇条書きに整形しています。
- **LLMプロンプトへのコンテキスト結合**
  - `src-tauri/src/ai.rs` の `refine_idea` および `generate_tasks_from_story` コマンドで引数 `project_id: String` を受け取るように関数シグネチャを修正しました。
  - 取得したMarkdown形式のプロジェクトデータを、LLMの System Prompt (System Instruction) の末尾に「現在のプロジェクトコンテキスト」として動的に結合しました。これにより、AIが現在のプロジェクトの進行状況や既存のスコープを把握した上で自律的に提案できるようになりました。
- **軽微な型エラーの修正**
  - `StorySwimlane.tsx` でタスク生成・追加時に `archived` プロパティの初期値を指定していなかったためのTypeScriptエラー（Epic 2.5 の残存修正）に対応し、`archived: false` を明示してエラーを取り除きました。

## テスト内容・結果 (Validation Result)

- **自動ビルド検証**:
  - `cargo check` による Rustレイヤーの静的解析エラーがゼロであることを確認済みです。
  - `npm run lint` によるTypeScriptレイヤーの静的検査において、新規実装したコンポーネントコードや型定義にエラーがないことを確認済みです。

## PO（人間）による確認手順 (Manual Testing Steps)

本番動作が意図通りに動くかどうかを、以下の手順で手動テストをお願いします：

1. アプリケーションで任意のプロジェクトを開き、いくつかテスト用のStoryとTaskを作成（または既存データを使用）します。
2. ヘッダーの💡アイコンをクリックし、「AI要件定義アシスタント」に対して次のような質問を投げかけます。
   - 「既存のストーリーと機能が重複しない、新しいアイデアを提案して」
3. LLMのアシスタントが、あなたの作成した既存のStory・Taskの内容を踏まえ、「すでにXXがありますので、それを拡張するYYはいかがですか？」のような文脈に沿った回答を行うことを確認してください。
4. ※コンテキスト自体は長すぎないように切り詰めていますが、もし出力の精度やコンテキストの量に違和感がある場合はレトロスペクティブ等でフィードバックしてください。
