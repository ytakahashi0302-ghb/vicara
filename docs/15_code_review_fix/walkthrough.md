# Walkthrough: Epic 2.5 コードベースの堅牢化

POの指示に基づき、CODE_REVIEW.md の指摘事項のうちCritical（バグ・アーキテクチャ負債）およびMedium（リファクタリング）1点を中心に改修を完了しました。

## 実装した変更内容

### 1. 外部キー制約の有効化
- `src-tauri/src/lib.rs` 内にDB初期化時のフックを追加し、接続完了直後に `PRAGMA foreign_keys = ON;` を実行するようにしました。これにより、SQLiteレベルでプロジェクト削除時などのカスケード制約が正しく機能するようになりました。

### 2. `archived` カラムの導入と状態判別の変更
- `src-tauri/migrations/4_add_archived_column.sql` を作成し、`stories` および `tasks` テーブルに `archived BOOLEAN DEFAULT FALSE` を追加しました。
- `db.rs` の各種クエリ（`get_stories`, `get_tasks` など）の実装を修正し、アクティブ判定を `sprint_id IS NULL` から `archived = 0` に変更しました。これにより、将来のスプリントプランニング（スプリント割り当て済みだが未完了）の設計に対応可能となりました。
- フロントエンドの `Story`, `Task` 型定義ファイル (`index.ts`) に `archived: boolean` を追加しました（現在フロントエンドでのフィルタリングはバックエンドが担っているため、状態保持のみ行います）。

### 3. スプリントアーカイブ処理のトランザクション化とバグ修正
- `src-tauri/src/db.rs` の `archive_sprint` 処理を sqlx のトランザクション（`let mut tx = pool.begin().await;` から `tx.commit().await;`）でラップしました。途中でエラーが起きた場合はロールバックが行われ、データ不整合にならなくなりました。
- タスクが0件のストーリーがアーカイブされるバグについて、SQL条件式に `AND EXISTS (SELECT 1 FROM tasks WHERE tasks.story_id = stories.id)` を追加し、「タスクが1件以上存在し、かつすべて完了（archived=1）しているストーリーのみ」をアーカイブ対象とするように修正しました。

### 4. AIリファクタリング・依存関係の追加
- `src-tauri/src/ai.rs` にて、約20行の重複コードとなっていたAPIキー取得処理を `get_api_key_and_provider` ヘルパー関数に切り出しました。
- トランザクション処理やDB初期化待機で `sleep` などを利用するため、`Cargo.toml` に `tokio` (features = ["time"]) を追加し、ビルドエラーを修正しました。

### 5. Technical Debt の記録
- 今回のスコープから外した指摘（JSONパースの脆弱性、モデル名のハードコード、楽観的UIの適用など）を `Rule.md` の技術的負債（Tech Debt）セクションに記録しました。

## PO（人間）へのお願い：手動検証の観点

以下の操作をお試しいただき、UIおよびデータが正常に動作することをご確認ください：

1. **マイグレーション・通常操作（CRUD）**
   - 既存のプロジェクトやストーリーが消えていないか確認してください。
   - 新規ストーリー、タスクが通常通り追加・編集・削除できるか確認してください。
2. **スプリントの開始とアーカイブ（トランザクションとバグ修正の確認）**
   - 新しいスプリントタイマーを開始し、完了させてもエラーにならないか。
   - スプリントをアーカイブする際、**タスクを1件も持たないストーリーが消えずにバックログに残っていること**を確認してください。
   - 完了したタスクを持つストーリーのみが、アーカイブ済みに移動することを確認してください。
3. **AI機能の確認（リファクタリング後）**
   - 既存のストーリーから「AIでタスクを生成」をクリックし、正しくタスクが生成されるか確認してください。

問題なければこれでEpic 2.5はクローズとし、本来予定していたEpic 3のタスクに戻ることが可能です！
