## スクラムイベントの設計方針

通常スクラムの5イベントをソロ×AI向けに3つに再設計します。

```
通常スクラム（5イベント）         vicara（3イベント）
─────────────────────────────────────────────────────
スプリントプランニング        →   プランニング
バックログリファインメント    →   （プランニングに統合）
デイリースクラム             →   （カンバンで代替・省略）
スプリントレビュー           →   レビュー＋レトロ（セット）
レトロスペクティブ           →   （レビューに統合）
```

**削減の理由：**
- バックログリファインメントは壁打ちエージェントとの会話で自然に実現できる
- デイリースクラムはソロ開発では不要（カンバンで進捗が常に見える）
- レビューとレトロは完了時にまとめて実施するのが自然なリズム

## エージェント設計方針

### ロール設計の考え方

スクラムの「スクラムマスター」「開発チーム」を別エージェントとして独立させるのではなく、
**各イベントの進行役として機能を溶け込ませる**設計とします。

独立したエージェントを増やすと管理コストと会話の複雑さが増す割に、
ソロ開発では恩恵が薄いためです。

### 実装予定のエージェント

| エージェント | タイミング | 役割 |
|---|---|---|
| 壁打ちエージェント | プランニング時 | アイディア → User Story への対話的整理 |
| タスク分解エージェント | プランニング時 | User Story → Tasks の自動生成・粒度チェック |
| 実行エージェント | スプリント実行時 | タスクの自律実行（ファイル操作・コード生成など） |
| レビューエージェント | レビュー時 | 成果物の品質チェック・受け入れ基準の確認 |
| レトロエージェント | レトロ時 | 各担当の振り返り生成・改善提案 |

### スクラムマスター機能の扱い

SMの機能（プロセスの番人・障害の交通整理・次に何が必要かの整理）は、
各イベントの冒頭処理として組み込みます。

```
プランニング開始時
→ 前回レトロの改善提案を参照し、今スプリントの進め方を調整

スプリント実行中
→ タスクのブロッカーを検出したら人間に通知

レトロ時
→ スプリント全体を俯瞰して障害パターンを整理
```

## レトロスペクティブとDB設計

### 各エージェントが振り返る内容

```
タスク分解エージェント
├── 粒度が大きすぎたタスクはなかったか
├── 見積もりとのズレはどうだったか
└── 依存関係の見落としはなかったか

実行エージェント
├── 詰まったポイントはどこか
├── 想定外の作業が発生したか
└── 次回同じタスクならどう改善するか
```

### SQLiteテーブル設計（予定）

```sql
CREATE TABLE agent_retrospectives (
  id            INTEGER PRIMARY KEY,
  sprint_id     INTEGER NOT NULL,
  agent_role    TEXT NOT NULL,      -- 'task_decomposer' | 'executor' | 'reviewer'
  good          TEXT,               -- うまくいったこと
  bad           TEXT,               -- うまくいかなかったこと
  next_action   TEXT,               -- 次スプリントへの改善アクション
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sprint_id) REFERENCES sprints(id)
);
```

### 次スプリントへの自動反映

プランニング開始時に直近Nスプリント分の`next_action`を取得し、
タスク分解・実行の各エージェントのコンテキストに自動注入します。

```
「前回レトロより：
  - タスク分解エージェント：認証まわりのタスクは細かく分解すること
  - 実行エージェント：外部APIを使うタスクはモックを先に作ること」
```

これにより、スプリントを重ねるほど各エージェントの精度が上がり、
**あなたの開発スタイルに合ったAIチームへと育っていきます。**

## 実行エージェントの承認フロー設計

### 基本方針：毎回確認しない、でも危険な操作は止まる

「全自動」と「毎回確認」の二択ではなく、**スプリント開始時の一括承認 × 危険度による自動分類**を組み合わせます。

毎回確認が必要なツール（Antigravityなど）は使い勝手が悪い一方、
Claude Code desktopのように信頼の文脈が蓄積されたツールは快適に使えます。
この快適さを、スクラムの設計で実現します。

### スプリント開始時の操作スコープ宣言

プランニング完了時にAIが今スプリントで行う操作範囲を提示し、人間が一度だけ承認します。

```
「今スプリントでは以下の操作を行います

  ✅ src/components/ 以下のファイル編集
  ✅ src/hooks/ 以下のファイル新規作成
  ✅ package.json への依存パッケージ追加
  ❌ データベーススキーマの変更は行いません
  ❌ src/ 外のファイルには触りません

  この範囲で進めてよいですか？」
```

**承認後はスプリント中、宣言範囲内を全自動で実行。**
スプリントという単位が、承認スコープの単位を兼ねます。

### 操作の危険度分類

宣言範囲内であっても、操作の危険度に応じて挙動を変えます。

```
危険度 LOW  → 自動実行（ログのみ記録）
  ├── ファイル読み取り
  ├── 宣言済みディレクトリへの新規ファイル作成
  └── プロジェクト内への書き込み

危険度 MID  → 自動実行 ＋ カンバンに通知バッジ表示
  ├── 既存ファイルの編集
  └── 依存パッケージの追加

危険度 HIGH → 承認ゲート（必ず止まる）
  ├── ファイル・ディレクトリの削除
  ├── 宣言範囲外へのアクセス
  └── 外部サービスへの書き込み系API呼び出し
```

### 操作ログとロールバック

実行エージェントの全操作はSQLiteに記録し、スプリント単位でのロールバックを可能にします。

```sql
CREATE TABLE agent_operations (
  id            INTEGER PRIMARY KEY,
  sprint_id     INTEGER NOT NULL,
  task_id       INTEGER NOT NULL,
  operation     TEXT NOT NULL,    -- 'file_write' | 'file_delete' | 'command' など
  target        TEXT NOT NULL,    -- 対象ファイルパス・コマンド
  risk_level    TEXT NOT NULL,    -- 'low' | 'mid' | 'high'
  approved_by   TEXT NOT NULL,    -- 'auto' | 'user'
  snapshot      TEXT,             -- 変更前の内容（ロールバック用）
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sprint_id) REFERENCES sprints(id),
  FOREIGN KEY (task_id) REFERENCES tasks(id)
);
```

## コスト管理設計

### ストーリーポイントではなく予算でベロシティを管理する

通常スクラムのベロシティ管理はストーリーポイントを使いますが、
AIにとっての「キャパシティ」はトークン＝お金です。
vicara では**スプリント予算（円）をベロシティの単位**として採用します。

```
通常スクラムのベロシティ管理
  前回スプリント：30SP消化
  今回スプリント：30SP分のタスクを選ぶ

vicara のベロシティ管理
  前回スプリント：¥180使用
  今回スプリント：¥200以内で進める
```

この発想はshyamagu氏の実験レポート（[AIエージェントだけでスクラムを回してみた](https://qiita.com/shyamagu/items/0ff827a1c2deef68ba9a)）でも言及されており、「AIに時間という概念が適切なのか微妙」という問いへの一つの答えでもあります。

### エージェントごとのモデル使い分け

全エージェントを高性能モデルで動かす必要はありません。
タスクの複雑さに応じてモデルを使い分け、コストを最適化します。

| エージェント | 推奨モデル | 理由 |
|---|---|---|
| 壁打ちエージェント | Opus | 文脈理解・対話品質が命 |
| レトロエージェント | Opus | 複雑な分析・改善提案が必要 |
| タスク分解エージェント | Sonnet | 構造化タスクは十分な性能 |
| 実行エージェント | Sonnet | コード生成・ファイル操作 |
| ログ記録・分類処理 | Haiku | 単純な処理はコスト最小化 |

### 2軸の制約設計：時間と予算

スプリントには**時間**と**予算**の2つの制約を設けます。
制約の種類が異なるため、どちらか一方では不十分です。

```
予算（¥）→ コストの上限。超えたら困る。AIへの依存度を制御する。
時間（h）→ 人間のリズム。1時間後に成果を見たい。区切りを作る。
```

どちらかに達した時点でスプリント完了とします。

### プランニング時の2軸承認フロー

スプリント開始前に推定コストと時間を提示し、操作スコープと同時に人間が承認します。

```
「今スプリントの制約を設定してください

  ⏱️ 時間上限  [ 1時間 ]   ← 人間のリズム
  💰 予算上限  [ ¥200  ]   ← コストの上限

  推定コスト内訳：
    壁打ち（Opus × 2回）        ¥ 40
    タスク分解（Sonnet × 1回）  ¥ 10
    実行（Sonnet × 8回）        ¥ 80
    レトロ（Opus × 1回）        ¥ 20
    ──────────────────────────
    推定合計                    ¥150〜200

  どちらかに達したらスプリント完了とします。開始しますか？」
```

### スプリント実行中の2軸表示

カンバン画面に時間と残予算をリアルタイムで並べて表示します。

```
⏱️ 経過時間  [38分 / 60分]  ████████░░░░  63%
💰 残予算    [¥143 / ¥200]  ████████░░░░  71%
```

2軸の状況によって意味が異なります：

```
先に時間が来たとき
→ 今日はここまで。続きは次のスプリントへ。

先に予算が来たとき
→ まだ時間はあるが、AIに頼りすぎた。
  残りは自分でやるか、次のスプリントに回す。

両方余ったとき
→ 予定より早く・安く終わった。ベロシティ改善の証拠。
```

### レトロへの2軸振り返りの組み込み

時間効率とコスト効率の両方をレトロの振り返り項目に含めます。
スプリントを重ねるほど改善傾向が可視化されます。

```
今スプリント
  ⏱️ 時間効率：45分 / 60分   → 15分余った
  💰 コスト効率：¥160 / ¥200 → ¥40余った

前回比：時間 -10分、コスト -¥20 → 改善傾向 📈
```

```sql
-- agent_retrospectives テーブルにコスト・時間列を追加
ALTER TABLE agent_retrospectives ADD COLUMN
  cost_actual   INTEGER,  -- 実際にかかったコスト（円）
  cost_budget   INTEGER,  -- スプリント予算（円）
  time_actual   INTEGER,  -- 実際にかかった時間（分）
  time_budget   INTEGER,  -- スプリント時間上限（分）
  cost_note     TEXT;     -- コスト・時間面での振り返り
                          -- 例：「Sonnetに切り替えて¥30削減、15分短縮」
```

### sprints テーブルへの2軸カラム追加

```sql
ALTER TABLE sprints ADD COLUMN
  budget        INTEGER,  -- スプリント予算上限（円）
  cost_actual   INTEGER,  -- 実績コスト（円）
  duration_min  INTEGER,  -- スプリント時間上限（分）
  time_actual   INTEGER;  -- 実績時間（分）
```
