# AI Scrum Tool (MicroScrum AI) 開発状況

## 現在の進捗・完了箇所
第9フェーズ（最終ポーリッシュ）および第9.5フェーズ（UXとローカライズ改善）を経てMVPが完了し、現在はv2.0に向けた開発フェーズ（Epic 1, 2）に入っています。

- **【完了】Epic 1: AI要件定義アシスタント**
  - AI要件定義アシスタントの機能（チャットUIとバックエンド連動）の実装を完了しました。

- **【完了】Epic 2: マルチプロジェクト管理（ワークスペース機能）**
  - **フェーズ1**: 既存のフロントエンド内でのDB直接操作（SQLiteプラグイン直叩き）を廃止し、Rust側（Tauriコマンド）への全面的な移行とアーキテクチャの整理を完了しました。さらに `tauri.conf.json` の機能を用いてDBを事前ロード（preload）する構成にリファクタリングしました。
  - **フェーズ2**: `projects` テーブルへのDBマイグレーション（SQLiteのテーブル再作成アプローチを利用）を実装し、プロジェクト管理のCRUDコマンド（Rust）を用意しました。また、既存モデル（Story, Task, Sprint）に `project_id` を持たせ、フロントの型定義も更新しました。
  - **フェーズ3**: `currentProjectId` を管理する `WorkspaceContext` を導入し、データ取得フックを動的化しました。さらに、ヘッダーに切り替えUI（ドロップダウン）と新規作成モーダルを実装し、リアクティブなプロジェクト切り替えを実現しました。また、スプリントタイマー稼働中にプロジェクトをまたがないようにする「Interaction Guard」と、タイマー状態の「プロジェクト別独立管理（`SprintTimerContext`経由）」を実装しました。

- **【完了】Epic 2.5: コードベースの堅牢化（技術的負債の返済）**
  - **トランザクションと外部キー制約**: `db.rs` においてデータベース接続時に `PRAGMA foreign_keys = ON;` を有効化し、`archive_sprint` 処理をトランザクション（`begin()` / `commit()`）でラップしてデータ不整合を防起しました。
  - **アーキテクチャの修正（`archived` フラグの導入）**: マイグレーションを実行して `stories` および `tasks` テーブルに `archived` カラム（BOOLEAN型）を追加しました。これに伴い、「アクティブ状態」の判別を `sprint_id IS NULL` から `archived = 0` へ変更しました。
  - **バグ修正**: スプリントアーカイブ時に「タスクが0件のストーリー」が意図せずアーカイブされてしまう問題のSQL条件式を修正しました。
  - **リファクタリング**: `ai.rs` にて、重複していたAPIキー取得のロジックを `get_api_key_and_provider` ヘルパー関数として切り出しました。
  - **引き継ぎ完了**: POによる検証が完了し、本Epicの対応は全てクローズしました。また、未対応の指摘（プロジェクト削除UIの欠如など）は全て `Rule.md` にTechnical Debtとして記録済みです。

## 次期セッションで着手すべきタスク（Next Steps）

コードベースが堅牢化し、技術的負債が整理されたため、いよいよEpic 3の新機能実装に着手可能です。

- **【完了】Epic 3: コンテキスト・アウェアなAI要件定義（簡易RAGの導入）**
  - **目標**: 選択中のプロジェクトの前提知識や既存ストーリーをLLMのコンテキストに結合し、文脈に沿った仕様提案を行う機能を実装する。
  - **完了内容**: 既存の `Story` および `Task` テーブルから `project_id` に属する有効なデータ (`archived = 0`) を取得し、動的かつ簡潔に整形されたMarkdownテキストをLLMの System Prompt の末尾に結合しました。これにより、AIアシスタント機能（壁打ち、タスク自動分解）がプロジェクト固有の既存文脈を把握できるようになり、重複提案を回避しつつ高度な要件定義提案が可能になりました。

### 1. Future Backlog（将来の構想・Epic 4以降）
